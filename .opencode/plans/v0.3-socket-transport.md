# Plan: v0.3 — Unix Socket Transport

## Goal

Add an optional Unix socket transport to `@attamusc/opencode-cmux` that
eliminates per-call child process spawning overhead. The CLI transport remains
functional, and a new `transport` config option controls selection. Default
behavior (`"auto"`) uses the socket when available, falling back to CLI.

---

## Context

### Current transport (v0.1–v0.2)

Every cmux call spawns a child process via `node:child_process.spawn()`:

```
coordinator → CliCmuxClient.execute() → spawn("cmux", [...args]) → wait for exit
```

This works reliably but incurs ~20–50ms per call for process creation, which
adds up during high-frequency tool tracking (Phase 1/v0.2) and sidebar log
updates (Phase 6/v0.2 rate-limits to 5/sec but that's still 5 spawns/sec).

### cmux socket protocol (confirmed from official docs at cmux.dev/docs/api)

The cmux socket at `/tmp/cmux.sock` uses **two distinct protocols** depending
on the command category:

**JSON-RPC** — for workspace, notification, split, input, and utility commands:
```json
{"id":"req-1","method":"notification.create","params":{"title":"Done","body":"Complete"}}
// Response:
{"id":"req-1","ok":true,"result":{...}}
```

**Legacy text format** — for all sidebar metadata commands (`set_status`,
`clear_status`, `set_progress`, `clear_progress`, `log`, `clear_log`, etc.):
```
set_status build compiling --icon=hammer --color=#ff9500 --tab=<workspace-uuid>
set_progress 0.5 --label=Building... --tab=<workspace-uuid>
log --level=error --source=build --tab=<workspace-uuid> -- Compilation failed
clear_status build --tab=<workspace-uuid>
clear_progress --tab=<workspace-uuid>
```

**Key observations (verified against official docs 2026-03-09):**
- JSON-RPC requests use `method`/`params`. Legacy v1 JSON payloads (`{"command":"..."}`) are not supported.
- Sidebar text commands use underscores (`set_status`, `clear_status`) not hyphens.
- Sidebar text commands use `--tab=<uuid>` for workspace scoping (not `--workspace`).
- Sidebar text commands use `--key=value` format (not `--key value` like the CLI).
- The `notify` command uses JSON-RPC method `notification.create`.
- The response format is **not** standard JSON-RPC 2.0 — it uses `{"id":...,"ok":true,"result":{...}}`
  with an `ok` boolean field rather than `jsonrpc:"2.0"` + `result`/`error`.
- Socket access modes: `off` (disabled), `cmuxOnly` (default, only cmux children), `allowAll`.
- The Python example in docs uses connect-per-call: `with socket.socket(...) as sock`.

### Architecture starting point

```
src/cmux/
  detect.ts      — CmuxEnvironment with socketPath, workspaceID, isManagedWorkspace
  commands.ts    — Pure CLI arg-array builders (buildNotifyCommand, buildSetStatusCommand, etc.)
  client.ts      — CliCmuxClient (spawn-based), createCmuxClient() factory
src/types.ts     — CmuxClient interface (notify, setStatus, clearStatus, setProgress, clearProgress, log)
src/config.ts    — PluginConfig (no transport option yet)
src/index.ts     — Plugin entrypoint, wires config + detect + client + coordinator
```

### Key constraints

- **Zero dependencies** — Socket client must use `node:net` built-ins only.
- **Bun runtime** — Build with `bun build`, test with `bun test`.
- **CmuxClient interface** — The socket client must implement the existing
  `CmuxClient` interface. Both transports are interchangeable. A new
  `readonly transport: "cli" | "socket"` property will be added to
  `CmuxClient` for discrimination (no `instanceof` checks needed).
- **Dual protocol** — The socket client must encode sidebar metadata commands in
  text format and notification commands in JSON-RPC format.

### Phase 2: Socket Client

**Goal:** Implement `SocketCmuxClient` — connect-per-call transport over the
Unix socket. Implements the same `CmuxClient` interface as `CliCmuxClient`.

#### 2a. Socket connection helper

**File: `src/cmux/socket-client.ts`** *(new)*

Create a low-level `socketRequest(options)` function:

```typescript
import { connect } from "node:net"

interface SocketRequestOptions {
  socketPath: string
  payload: string     // newline-terminated string to send
  timeoutMs: number
}

interface SocketResult {
  response: string
  error?: undefined
}

interface SocketError {
  response?: undefined
  error: { code: string; message: string }
}

type SocketOutcome = SocketResult | SocketError

function socketRequest(options: SocketRequestOptions): Promise<SocketOutcome>
```

Behavior:
1. `net.connect(socketPath)`
2. Write `payload` (already `\n`-terminated from builders)
3. Collect response data chunks
4. Resolve with response string on `end` event
5. On `error`: resolve with `SocketError` (never reject/throw)
6. On `timeout`: `socket.destroy()`, resolve with timeout error

The function is connect-per-call — each invocation opens and closes a socket.
This is simpler than persistent connections and already much faster than
spawning processes (~1-2ms vs ~20-50ms). Persistent connections can be a future
optimization if profiling warrants it.

**Important:** `socketRequest` never rejects. All errors are returned as
`SocketError` outcomes. This makes the calling code simpler — no try/catch
needed, just pattern match on the result.

#### 2b. SocketCmuxClient class

**File: `src/cmux/socket-client.ts`**

```typescript
interface SocketCmuxClientOptions {
  socketPath: string
  workspaceID?: string
  logger: PluginLogger
  timeoutMs?: number  // default: 5000
}

export class SocketCmuxClient implements CmuxClient {
  public readonly available: boolean
  public readonly transport = "socket" as const
  public readonly workspaceID?: string
  private requestCounter = 0
  private reportedConnectionFailure = false

  constructor(options: SocketCmuxClientOptions) { ... }

  // CmuxClient methods — each delegates to socketRequest()
  async notify(payload: NotificationPayload): Promise<void>
  async setStatus(key: string, payload: SidebarStatusPayload): Promise<void>
  async clearStatus(key: string): Promise<void>
  async setProgress(payload: ProgressPayload): Promise<void>
  async clearProgress(): Promise<void>
  async log(payload: SidebarLogPayload): Promise<void>
}
```

**Method encoding strategy:**
- `notify()` → `buildSocketNotify()` (JSON-RPC) → parse response, check `ok`
- `setStatus()` → `buildSocketSetStatus()` (text format) → fire-and-forget
- `clearStatus()` → `buildSocketClearStatus()` (text format) → fire-and-forget
- `setProgress()` → `buildSocketSetProgress()` (text format) → fire-and-forget
- `clearProgress()` → `buildSocketClearProgress()` (text format) → fire-and-forget
- `log()` → `buildSocketLog()` (text format) → fire-and-forget

**JSON-RPC response handling (for `notify`):**
```typescript
private async sendJsonRpc(payload: string, label: string): Promise<void> {
  const outcome = await socketRequest({ ... })
  if (outcome.error) {
    this.handleError(outcome.error, label)
    return
  }
  const parsed = parseCmuxResponse(outcome.response)
  if (parsed && !parsed.ok) {
    await this.options.logger.log("warn", `cmux ${label} returned error`, {
      error: parsed.error,
    })
  }
}
```

**Text-format handling (for sidebar commands):**
```typescript
private async sendText(payload: string, label: string): Promise<void> {
  const outcome = await socketRequest({ ... })
  if (outcome.error) {
    this.handleError(outcome.error, label)
  }
  // Response ignored for text-format commands
}
```

**Error handling** (mirrors `CliCmuxClient` patterns from `src/cmux/client.ts:102-136`):
- `ECONNREFUSED` / `ENOENT` → log once via `reportedConnectionFailure` flag,
  then silently no-op (same pattern as `CliCmuxClient.reportedMissingBinary`)
- Timeout (5s default) → log warning, resolve without throwing
- Malformed response → log warning, don't crash

**`available` property:** Set to `true` in constructor. The socket client is
created only when the socket path exists, so `available` is always true. If a
call fails at runtime, individual operations no-op gracefully.

**Request ID generation:** `private nextRequestID(): string` returns
`"req-${++this.requestCounter}"`. String IDs for consistency with cmux docs.

function parseTransport(value: string | undefined, fallback: "cli" | "socket" | "auto"): "cli" | "socket" | "auto" {
  if (!value) return fallback
  const normalized = value.trim().toLowerCase()
  if (normalized === "cli" || normalized === "socket" || normalized === "auto") return normalized
  return fallback
}

```typescript
import { statSync } from "node:fs"

function checkSocketExists(socketPath: string): boolean {
  try {
    const stat = statSync(socketPath)
    return stat.isSocket()
  } catch {
    return false
  }
}
```

This is a one-time synchronous check at startup — not on every call. The
official cmux docs recommend `[ -S "$SOCK" ]` which is the shell equivalent.

#### 3d. Transport factory

**File: `src/cmux/client.ts`**

Modify `CreateCmuxClientOptions` and `createCmuxClient()`:

```typescript
interface CreateCmuxClientOptions {
  binary: string
  environment: CmuxEnvironment
  logger: PluginLogger
  transport: "cli" | "socket" | "auto"
}

export function createCmuxClient(options: CreateCmuxClientOptions): CmuxClient {
  const useSocket = shouldUseSocket(options.transport, options.environment, options.logger)

  if (useSocket) {
    return new SocketCmuxClient({
      socketPath: options.environment.socketPath,
      workspaceID: options.environment.workspaceID,
      logger: options.logger,
    })
  }

  return new CliCmuxClient(options)
}

function shouldUseSocket(
  transport: string,
  env: CmuxEnvironment,
  logger: PluginLogger,
): boolean {
  if (transport === "cli") return false

  if (transport === "socket") {
    if (!env.hasSocket) {
      // User explicitly requested socket but it's not available
      logger.log("warn", "Socket transport requested but socket not found, falling back to CLI", {
        socketPath: env.socketPath,
      })
      return false
    }
    return true
  }

  // "auto" — use socket if available, silent fallback to CLI
  return env.hasSocket
}
```

Note: `shouldUseSocket` is synchronous in its decision but may call `logger.log`
(which is async). The logger call is fire-and-forget since this is a warning.

## Resolved Questions (from research)

### RQ1: Text-format value escaping

**Status: Partially resolved.** The official cmux docs show `--label=Building...`
without spaces. The `log` command uses `-- Compilation failed` where everything
after `--` is the rest of the line. Phase 0 will verify multi-word labels at
runtime. Start without quoting. The `log` message body is safe because it uses
the `--` separator convention.

### RQ2: Text-format response shape

**Status: Resolved.** The official docs don't document responses for text-format
commands. The Python client example only reads responses for JSON-RPC calls.
**Decision:** Treat text-format commands as fire-and-forget. Read whatever
comes back on the socket but don't parse it. Only parse JSON-RPC responses
(for `notify`) and check the `ok` field.

### RQ3: Socket file detection reliability

**Status: Resolved.** The official docs recommend `[ -S "$SOCK" ]` (file
existence check). Using `statSync` + `isSocket()` at startup is the standard
approach. If cmux restarts, the plugin must be restarted too. A file watcher
can be added in a future version.

### RQ4: Persistent connections

**Status: Deferred.** Connect-per-call is the starting design. Unix socket
connect is typically <1ms. The official Python example also uses connect-per-call.
No change needed.

### RQ5: Workspace ID format

**Status: RESOLVED.** `CMUX_WORKSPACE_ID` contains **UUIDs** (e.g.,
`C741C8F0-DD75-4BF2-83BF-2CC032234753`), not workspace refs. Passing the
UUID directly to `--tab=` works for all sidebar text-format commands. No
`system.identify` or `workspace.current` fallback needed.

### RQ6: Response format (NEW)

**Status: Resolved.** The cmux socket uses a custom response format
`{"id":"...","ok":true,"result":{...}}`, NOT standard JSON-RPC 2.0. The
`parseCmuxResponse` function in Phase 1 handles this. For `ok:false` responses
(error case), log a warning with the error details. Don't throw.

### RQ7: CmuxClient transport discrimination (NEW)

**Status: Resolved.** Added `readonly transport: "cli" | "socket"` to the
`CmuxClient` interface. No `instanceof` checks needed. The property is set
as a const assertion in each implementation class.

## Risks

1. **Dual-protocol complexity** — The socket client must handle two encoding
   formats. This is the main complexity risk. Mitigation: the encoding is
   isolated in pure functions (Phase 1) with thorough test coverage, separate
   from I/O concerns.

2. **Socket access mode** — cmux has access modes (`off`, `cmuxOnly`,
   `allowAll`). If the socket is in `off` mode or the process isn't recognized
   as a cmux child, connections will be refused. The client handles this
   gracefully (ECONNREFUSED → log once, no-op), but users may be confused.
   The README should mention that socket transport requires the socket to be
   enabled.

3. **Bun `node:net` compatibility** — Bun's `node:net` implementation should
   be mature enough for Unix domain sockets, but edge cases in timeout handling
   or error codes could differ from Node.js. Tests run in Bun, so any issues
   surface early.

4. **Workspace ref vs UUID** — Phase 0 verifies whether `--tab=workspace:1`
   works. If it doesn't, Phase 2 needs a `resolveWorkspaceUUID()` helper that
   calls `system.identify` or `workspace.current` via JSON-RPC at construction
   time. This is a bounded scope increase (one additional JSON-RPC call at
   startup, cached result).
