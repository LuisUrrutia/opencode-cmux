# Plan: v0.4 — Maintainability & Agentic Friendliness

## Goal

Harden `@attamusc/opencode-cmux` for long-term maintainability and make it
easy for agentic coding loops to work on the codebase safely. This release
fixes latent bugs, plugs memory/lifecycle leaks, adds documentation for the
implicit behavioral contracts, and improves test coverage — without changing
any user-visible behavior.

---

## Context

### Current state (v0.3)

The plugin is feature-complete: session tracking, tool/file/todo feedback,
question/permission overlays, progress bars, notifications, sidebar logs,
env-var config, and a dual CLI/socket transport. The code is clean and
well-structured, but an advisor review surfaced 12 concrete issues across
four categories:

1. **Latent bugs** — Unprotected async callbacks, permanent caching of
   transient failures, unescaped multi-word socket payloads.
2. **Memory/lifecycle leaks** — Unbounded `lastFileEditAt` map, missing
   `dispose()` call from the plugin entry point.
3. **Missing documentation** — No AGENTS.md, no JSDoc on the core state
   priority contract or config fields.
4. **Test gaps** — No coverage for error/permission paths, test helpers
   inlined in a single file.

### Key constraints

- **No behavioral changes** — All fixes must be invisible to end users.
  Existing tests must continue to pass unchanged (unless they're testing
  behavior that was already wrong).
- **Zero dependencies** — Stays zero-dep.
- **Bun runtime** — `bun build` and `bun test` remain the only build/test
  commands.
- **Backward-compatible config** — No new config options in this release.

### Phase 2: Fix Latent Bugs

**Goal:** Fix three bugs that could cause silent failures or incorrect
behavior in production. These are all small, surgical changes.

#### 2a. Wrap async setTimeout callbacks in try/catch

**File: `src/state/presenter.ts`**

The render timer (line 584) and stale session watchdog (line 637) both have
`async () => { ... }` callbacks passed to `setTimeout`. If either throws,
the rejection is unhandled — violating the "never crash the host" contract.

**Before (line 584):**
```typescript
this.renderTimer = setTimeout(async () => {
  this.renderPending = false
  this.renderTimer = undefined
  await this.renderNow()
}, RENDER_THROTTLE_MS - elapsed)
```

**After:**
```typescript
this.renderTimer = setTimeout(async () => {
  try {
    this.renderPending = false
    this.renderTimer = undefined
    await this.renderNow()
  } catch (err) {
    this.options.logger.log("error", `Deferred render failed: ${err}`)
  }
}, RENDER_THROTTLE_MS - elapsed)
```

Apply the same pattern to the stale watchdog at line 637:
```typescript
this.staleTimer = setTimeout(async () => {
  try {
    // ... existing logic ...
  } catch (err) {
    this.options.logger.log("error", `Stale timer failed: ${err}`)
  }
}, timeoutMs)
```

**Estimated effort:** 10 minutes.

#### 1b. Don't cache fallback session metadata permanently

**File: `src/opencode/session-resolver.ts`**

Lines 26 and 50 both cache error/unavailable fallback metadata with
`kind: "primary"`. If the API is transiently down, a subagent session gets
permanently misclassified as primary — affecting notification routing and
sidebar labels for the rest of the plugin lifetime.

**Fix:** Skip caching when the result is a fallback. The next call will
retry the API, and on success the real metadata gets cached normally.

**Before (line 26):**
```typescript
const fallback = {
  id: sessionID,
  title: sessionID,
  kind: "primary" as const,
}
this.cache.set(sessionID, fallback)
```

**After:**
```typescript
const fallback = {
  id: sessionID,
  title: sessionID,
  kind: "primary" as const,
}
// Don't cache — transient failures should be retried
return fallback
```

Apply the same fix to the catch block at line 50. Both `this.cache.set()`
calls for fallbacks should be removed.

**Consideration:** This means every call for an unknown session re-hits the
API when it's down, potentially generating many warn-level log lines. To
mitigate, add a short negative TTL (e.g. 5 seconds) using a separate
`failedAt` map:

```typescript
private failedAt = new Map<string, number>()
private static readonly NEGATIVE_CACHE_TTL_MS = 5_000

public async getSessionMetadata(sessionID: string): Promise<SessionMetadata> {
  const cached = this.cache.get(sessionID)
  if (cached) return cached

  // Don't retry too quickly after a failure
  const lastFail = this.failedAt.get(sessionID)
  if (lastFail && Date.now() - lastFail < OpencodeSessionResolver.NEGATIVE_CACHE_TTL_MS) {
    return { id: sessionID, title: sessionID, kind: "primary" }
  }
  // ... rest of resolve logic ...
}
```

In the catch block, set `this.failedAt.set(sessionID, Date.now())` instead
of `this.cache.set(...)`.

**Estimated effort:** 20 minutes.

#### 1c. Escape multi-word values in socket text protocol

**File: `src/cmux/commands.ts`**

Line 89 builds `set_status ${key} ${payload.text}` which breaks when
`payload.text` contains spaces (e.g. `"working: bash · 1 subagent"`). The
cmux text protocol expects space-separated positional args, so multi-word
values must be quoted.

**Before (line 89):**
```typescript
const command = `set_status ${key} ${payload.text} --icon=${payload.icon} --color=${payload.color}`
```

**After:**
```typescript
function quote(value: string): string {
  return value.includes(" ") ? `"${value}"` : value
}

const command = `set_status ${key} ${quote(payload.text)} --icon=${payload.icon} --color=${payload.color}`
```

Review all other `buildSocket*` functions in commands.ts for similar issues.
The `buildSocketLog` function likely has the same problem with its message
body.

**Estimated effort:** 15 minutes.

### Phase 2: Memory & Lifecycle Fixes

**Goal:** Fix the unbounded `lastFileEditAt` map and wire up `dispose()`
from the plugin entry point so timers don't leak.

#### 2a. Bound the `lastFileEditAt` map

**File: `src/state/presenter.ts`**

Line 77: `private readonly lastFileEditAt = new Map<string, number>()`
never evicts entries. Over a long session with many file edits, this grows
without bound. The `recentFiles` ring buffer (max 5) is correctly bounded
but the debounce map isn't.

**Fix:** After every file edit, prune entries older than the debounce window:

```typescript
private pruneStaleFileEdits(): void {
  const cutoff = Date.now() - FILE_EDIT_DEBOUNCE_MS
  for (const [path, timestamp] of this.lastFileEditAt) {
    if (timestamp < cutoff) {
      this.lastFileEditAt.delete(path)
    }
  }
}
```

Call `pruneStaleFileEdits()` at the end of `handleFileEdited()`. This keeps
the map bounded to at most one entry per unique file edited within the
debounce window.

**Estimated effort:** 10 minutes.

#### 2b. Wire up `dispose()` from the plugin entry point

**File: `src/index.ts`**

The coordinator has a `dispose()` method that clears the render timer and
stale watchdog, but `index.ts` never calls it. If the plugin is unloaded,
stale `setTimeout` references keep the coordinator alive.

**Fix:** Return `dispose` in the hook object. Check if OpenCode's plugin
contract supports a `dispose` or `cleanup` hook. If it does:

```typescript
return {
  async event({ event }) { /* ... */ },
  async "permission.ask"(input) { /* ... */ },
  async "tool.execute.before"(input, output) { /* ... */ },
  async "tool.execute.after"(input, output) { /* ... */ },
  dispose() {
    coordinator.dispose()
  },
}
```

If OpenCode doesn't support a `dispose` hook, document this limitation in
AGENTS.md with a note that timers self-clear when the session goes idle
(the stale timer only runs while busy, and the render timer is one-shot
per throttle cycle).

**Investigation needed:** Check whether OpenCode's plugin system calls a
`dispose()` or similar lifecycle method when the plugin is unloaded.

**Estimated effort:** 15 minutes (including investigation).

```markdown
# AGENTS.md

## Mental Model

This is an OpenCode plugin that bridges OpenCode's event hooks to cmux's
sidebar/notification UI. The core loop is:

  OpenCode events -> normalize -> CmuxStateCoordinator -> cmux commands

The coordinator is a state machine with an implicit priority order for
what gets displayed in the sidebar status pill.

## State Priority Order (highest to lowest)

1. Permission overlay (blocks everything)
2. Question overlay
3. Error state
4. Session busy (with tool/file/todo detail)
5. Session idle / done

An agent modifying `buildSnapshot()` MUST preserve this priority order.

## How to Add a New Event Type

1. Add variant to `NormalizedEvent` union in `src/events.ts`
2. Add `case` in `normalizeEvent()` in the same file
3. Add `handle*()` method on `CmuxStateCoordinator` in `src/state/presenter.ts`
4. Wire the hook in `src/index.ts` (event switch or named hook)
5. Add tests in `tests/presenter.test.ts`

## Constraints

- Zero dependencies — use Node.js/Bun built-ins only
- Never throw from hook handlers — errors must be caught and logged
- The plugin must be invisible when cmux is not detected
- Socket text protocol: positional args are space-separated, multi-word
  values must be quoted

## Config

All config is via environment variables. See `src/config.ts` for the full
list. The `PluginConfig` interface in `src/types.ts` defines the shape.

## Testing

- `bun test` — runs all tests
- Test helpers (FakeCmuxClient, FakeSessionResolver, noopLogger) are in
  `tests/helpers/` (after Phase 4)
```

## Non-goals (deferred)

These were identified by the advisor review but are not worth doing in v0.4:

- **Coordinator decomposition** — Extracting `SessionStateManager` and
  `SidebarRenderer` from the 725-line `CmuxStateCoordinator` would improve
  separation of concerns, but the coordinator is stable and well-tested.
  Decomposition adds risk for marginal benefit unless we're actively adding
  major features. Revisit if the file grows past ~1000 lines.

- **`activeTools` cleanup on session end** — The `activeTools` map could
  theoretically retain stale entries if `tool.execute.after` is never
  called. In practice OpenCode always fires the after hook. Adding cleanup
  in `handleSessionIdle` is cheap but adds complexity for a theoretical
  issue. Revisit if tool tracking bugs are reported.
