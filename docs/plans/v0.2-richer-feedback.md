# Plan: v0.2 — Richer Agent Feedback

## Goal

Evolve `@attamusc/opencode-cmux` from coarse session-level status reporting to
granular, real-time agent activity feedback — including tool execution tracking,
file activity, todo state, better progress estimation — and add an optional Unix
socket transport to eliminate per-call process spawning overhead.

---

## Context

### Current architecture (v0.1)

```
src/
  index.ts                     — Plugin entrypoint, dispatches events
  types.ts                     — All shared interfaces
  config.ts                    — Env-var config with boolean parsing
  events.ts                    — Normalizes raw events → NormalizedEvent union
  logger.ts                    — Thin wrapper around client.app.log
  cmux/
    detect.ts                  — Reads CMUX env vars, checks socket existence
    commands.ts                — Pure arg-array builders for each cmux subcommand
    client.ts                  — CliCmuxClient: spawns `cmux` child process
  opencode/
    session-resolver.ts        — client.session.get() with in-memory cache
  state/
    session-state.ts           — SessionRuntime type, formatSessionLabel, getBusySubagentCount
    project-context.ts         — Resolves project label/id/root from ctx
    presenter.ts               — CmuxStateCoordinator: state machine, snapshot diff, applies to cmux
```

**What works:** Session busy/idle/error, question/permission overlays, subagent
counting, status pills, progress bars, notifications, sidebar logs, env-var config.

**What's missing:** Tool execution visibility, file edit tracking, session
lifecycle events, todo tracking, progress estimation, socket transport, and the
events/hooks infrastructure to support all of this.

### Key constraints

- **cmux socket protocol:** JSON-RPC over `/tmp/cmux.sock`. One newline-terminated
  JSON request per call. Response: `{"id":"...","ok":true,"result":{...}}`. The
  sidebar metadata commands (`set-status`, `log`, etc.) use the legacy v1 text
  format over socket, not JSON-RPC `method`/`params` — confirmed by the docs
  showing `set_status build compiling --icon=hammer --color=#ff9500 --tab=<uuid>`
  for socket calls. **Investigate this further before implementing socket
  transport** — the notification commands use JSON-RPC but sidebar commands
  appear to use a different protocol.
- **OpenCode plugin hooks:** Two hook styles exist:
  1. `event({ event })` — receives `{ type, properties }` for all events
  2. Named hooks like `"permission.ask"(input, output)`, `"tool.execute.before"(input, output)`,
     `"tool.execute.after"(input, output)`, `"file.edited"(input, output)`, etc.
- **Bun runtime:** Build with `bun build`, test with `bun test`. No Node.js
  compatibility layer needed.
- **No dependencies** in package.json — plugin is zero-dep. Socket client must
  use Node.js/Bun built-ins (`node:net`).

## Phases

### Phase 1: Tool Execution Tracking

**Goal:** Show the user what tool the agent is currently running and log tool
completions to the sidebar. This is the highest-value improvement — it transforms
"working" into "running bash: npm test" or "editing src/index.ts".

#### 1a. Extend the event normalizer

**File: `src/events.ts`**

Add new variants to the `NormalizedEvent` union:

```typescript
| { type: "tool.started"; sessionID?: string; tool: string; args?: Record<string, unknown> }
| { type: "tool.completed"; sessionID?: string; tool: string; duration?: number; error?: string }
```

Subscribe to `tool.execute.before` and `tool.execute.after` via the plugin hooks
system. These are **named hooks**, not generic events — they receive
`(input, output)` not `{ event }`.

The `input` shape for tool hooks (from OpenCode plugin docs):
- `input.tool` — tool name (e.g., "bash", "read", "edit", "write", "glob", "grep")
- `output.args` — tool arguments (e.g., `{ command: "npm test" }` for bash, `{ filePath: "..." }` for read)

**File: `src/index.ts`**

Add named hook handlers alongside the existing `event` and `permission.ask` hooks:

```typescript
"tool.execute.before": async (input, output) => {
  await coordinator.handleToolStarted(input.tool, output?.args)
},
"tool.execute.after": async (input, output) => {
  await coordinator.handleToolCompleted(input.tool, output?.args)
},
```

**File: `src/types.ts`**

Add to `PluginHooks`:
```typescript
"tool.execute.before"?: (input: ToolExecuteInput, output?: ToolExecuteOutput) => Promise<void>
"tool.execute.after"?: (input: ToolExecuteInput, output?: ToolExecuteOutput) => Promise<void>
```

Add interfaces:
```typescript
interface ToolExecuteInput {
  tool: string
  [key: string]: unknown
}

interface ToolExecuteOutput {
  args?: Record<string, unknown>
  [key: string]: unknown
}
```

#### 1b. Track active tools in the coordinator

**File: `src/state/presenter.ts`**

Add tool tracking state to `CmuxStateCoordinator`:

```typescript
private activeTools = new Map<string, { tool: string; startedAt: number; args?: Record<string, unknown> }>()
```

Add methods:
- `handleToolStarted(tool: string, args?: Record<string, unknown>)` — Adds to
  `activeTools`, increments `toolCallCount`, logs to sidebar (e.g.,
  `"running bash: npm test"`), updates status text.
- `handleToolCompleted(tool: string, args?: Record<string, unknown>)` — Removes
  from `activeTools`, logs completion, updates status text.

#### 1c. Enrich status text with tool info

Add a helper `describeToolActivity()` that picks the most relevant tool to

### Phase 4: Better Progress Estimation

**Goal:** Replace static progress values (0.25 for busy, 0.66 for question) with
dynamic estimation based on observable signals.

#### 4a. Progress tracker module

**File: `src/state/progress-tracker.ts`** *(new)*

Create a `ProgressTracker` class that estimates progress based on:

1. **Tool call count** — More tool calls = further along. Use a logarithmic
   curve: `progress = 0.1 + 0.6 * (1 - 1 / (1 + toolCalls * 0.15))`
   This gives: 0 tools → 0.10, 5 tools → 0.41, 10 tools → 0.54, 20 tools → 0.62
2. **Time elapsed** — Slow ramp as fallback for sessions with few tool calls
3. **Todo completion** — If todos are present, weight towards `completed / total`
4. **Activity phase** — Questions/permissions clamp to ~0.66 (waiting), idle = 1.0

```typescript
export class ProgressTracker {
  private toolCalls = 0
  private startedAt?: number
  private todoTotal = 0
  private todoCompleted = 0

  recordToolCall(): void
  updateTodos(total: number, completed: number): void
  start(): void
  estimate(): number  // Returns 0.0–1.0
  reset(): void
}
```

The key insight: progress should never go backwards. The tracker must maintain
a high-water mark.

#### 4b. Integrate into coordinator

**File: `src/state/presenter.ts`**

Replace the hard-coded progress values in `buildSnapshot()` with calls to
`ProgressTracker.estimate()`:

- Remove: `value: 0.25` / `value: 0.35` / `value: 0.66`
- Replace with: `value: this.progressTracker.estimate()`
- Still clamp question/permission states to ≥ 0.50
- Idle state is always `1.0`

#### 4c. Config

**File: `src/config.ts`**

The existing `progressEnabled` toggle is sufficient. No new config needed.

### Phase 5: Unix Socket Transport

**Goal:** Add an optional socket transport that eliminates child process
spawning. CLI transport remains the default. Socket transport is opt-in via
config.

#### 5a. Investigate sidebar metadata socket protocol

**IMPORTANT: Pre-work required.** The cmux docs show two different socket
protocols:

1. **JSON-RPC** for workspace/notification/utility commands:
   `{"id":"...","method":"notification.create","params":{...}}`
2. **Legacy text format** for sidebar metadata commands:
   `set_status build compiling --icon=hammer --color=#ff9500 --tab=<uuid>`

Before implementing, manually verify:
- Does the socket accept JSON-RPC for sidebar commands? (i.e., is there a
  `sidebar.set_status` method?)
- Or must sidebar commands use the text format?
- Or does the text format apply to all commands?

Test with: `echo '{"id":"test","method":"system.ping","params":{}}' | nc -U /tmp/cmux.sock`

This investigation determines the socket client's encoding strategy.

#### 5b. Socket client

**File: `src/cmux/socket-client.ts`** *(new)*

Implement `SocketCmuxClient` implementing the same `CmuxClient` interface:

```typescript
import { connect } from "node:net"

export class SocketCmuxClient implements CmuxClient {
  private requestID = 0
  readonly available: boolean
  readonly workspaceID?: string

  constructor(options: SocketClientOptions) { ... }

  // Each method: open connection → write JSON-RPC → read response → close
  // Connections are short-lived (connect-per-call) to avoid reconnection complexity
  // in Phase 5. Persistent connections can be Phase 6 if perf warrants it.
}
```

**Connection strategy:** Connect-per-call (not persistent). Each cmux call:
1. `net.connect(socketPath)`
2. Write JSON line + `\n`
3. Read response
4. Close socket

This is simpler than persistent connections and already much faster than
spawning a child process. A persistent connection with reconnection can be
a future optimization.

**Error handling:**
- Socket not found → log once, fall back to no-op (same as CLI ENOENT)
- Connection refused → log, fall back
- Timeout → 5s per call, log warning

#### 5c. Transport factory

**File: `src/cmux/client.ts`**

Modify `createCmuxClient()` to accept a transport preference:

```typescript
export function createCmuxClient(options: CreateCmuxClientOptions): CmuxClient {
  if (options.transport === "socket" && options.environment.hasSocket) {
    return new SocketCmuxClient(options)
  }
  return new CliCmuxClient(options)
}
```

**File: `src/config.ts`**

```typescript
transport: "cli" | "socket" | "auto"   // OPENCODE_CMUX_TRANSPORT, default: "auto"
```

- `"cli"` — Always use CLI (current behavior)
- `"socket"` — Use socket, fall back to CLI if unavailable
- `"auto"` — Use socket if available, CLI otherwise

#### 5d. Shared command encoding

Both transports need to encode cmux commands. The CLI client already uses
`commands.ts` for arg arrays. The socket client needs either:

- JSON-RPC payloads (if sidebar commands support it)
- Text-format command strings (if they don't)

**File: `src/cmux/commands.ts`**

Add parallel functions for socket encoding, e.g.:
```typescript
export function buildNotifyRPC(payload: NotificationPayload, requestID: number): string
export function buildSetStatusText(key: string, payload: SidebarStatusPayload, workspaceID?: string): string
```

The exact approach depends on Phase 5a investigation results.

### Phase 6: Error Handling & Resilience Hardening

**Goal:** Ensure the plugin is bulletproof in degraded environments.

#### 6a. Graceful degradation for named hooks

Not all OpenCode versions may support all hooks. The plugin should:
- Wrap each named hook registration in a try/catch
- Log a debug message if a hook type isn't supported
- Continue working with whatever hooks are available

**File: `src/index.ts`**

Add runtime detection: if `tool.execute.before` isn't called after N session
events, log a debug note that tool tracking may not be available.

#### 6b. Rate limiting for cmux calls

During heavy tool usage (e.g., agent running many file reads), the plugin
could flood cmux with sidebar updates. Add throttling:

**File: `src/state/presenter.ts`**

- Throttle `render()` to max once per 200ms during rapid state changes
- Throttle sidebar log calls to max 5/second
- Deduplicate identical consecutive status updates (already done via snapshot diff)

#### 6c. Stale state cleanup

If a session goes silent (no events for 5+ minutes after being "busy"), it
may have crashed without firing an idle/error event. Add a watchdog:

**File: `src/state/presenter.ts`**

```typescript
private lastEventAt = Date.now()
```

On each event, update `lastEventAt`. Optionally (configurable), clear stuck
"working" states after a timeout. This is risky — could clear legitimate
long-running operations — so make it opt-in:

**File: `src/config.ts`**

```typescript
staleSessionTimeoutMs: number  // OPENCODE_CMUX_STALE_TIMEOUT, default: 0 (disabled)
```

#### 6d. Tests

**File: `tests/presenter.test.ts`**

- [ ] Rapid tool events are throttled (max render rate)
- [ ] Sidebar log rate limiting works
- [ ] Stale session cleanup fires after timeout (when enabled)
- [ ] Stale session cleanup doesn't fire when disabled

## Open Questions

1. **Tool hook `input`/`output` shapes** — The OpenCode plugin docs show
   `tool.execute.before` and `tool.execute.after` as named hooks but don't
   fully document the `input`/`output` parameter shapes. Need to verify at
   runtime (add debug logging in Phase 1 to dump the shapes, then tighten
   types).

2. **`file.edited` delivery mechanism** — Is it a generic event (comes through
   `event({ event })`) or a named hook (`"file.edited"(input, output)`)? The
   plugin docs list it under "Events" but examples only show `event.type`
   matching. Plan for both.

3. **`todo.updated` properties shape** — Not documented. Need to inspect at
   runtime. The plan assumes `{ items: [{ text, completed }] }` but this
   needs verification.

4. **cmux sidebar socket protocol** — The docs show JSON-RPC for
   workspace/notification commands but text format for sidebar metadata. Need to
   test whether JSON-RPC methods exist for sidebar commands before implementing
   the socket client.

5. **Should socket transport be persistent or connect-per-call?** — The plan
   starts with connect-per-call for simplicity. If latency testing shows this is
   too slow, a persistent connection with reconnect logic can be added later.

6. **Message events** — `message.part.updated` and `message.updated` could
   provide streaming token count or content length for progress estimation. Worth
   investigating but deferred to avoid scope creep.

## Risks

1. **OpenCode event shape instability** — The plugin system is still evolving.
   Hook signatures and event property shapes may change between versions. Mitigate
   with defensive parsing (the existing `getString` / `asRecord` pattern) and
   runtime shape logging.

2. **cmux sidebar command rate limits** — Rapid tool execution could generate
   many sidebar updates. The Phase 6 throttling is essential — consider pulling
   it into Phase 1 if tool events fire very frequently.

3. **Socket protocol ambiguity** — If sidebar metadata commands only work in text
   format over the socket, the socket client needs a dual-protocol encoder. This
   adds complexity. Worst case: use JSON-RPC for notifications and CLI for
   sidebar, or skip sidebar commands over socket entirely.

4. **Progress estimation accuracy** — Logarithmic tool-count curves are
   heuristic. Users may find the progress bar misleading if sessions vary wildly
   in length. Consider making progress estimation opt-in or clearly labeling it
   as approximate.

5. **Test isolation** — The coordinator has growing state. Tests need careful
   setup/teardown. Consider extracting a `createTestCoordinator()` factory with
   sensible defaults to reduce test boilerplate.
